# GoNow 多 Agent 协作开发架构方案

> 灵感来源：Simon（Reviewly 创始人）的「一人 + 五 Agent」模式
> 结合 Vibe Coding 方法论，基于 SOLO 子 Agent 调度能力实现
> 日期：2026-04-16 | 版本：V2.0（7 Agent 全激活版）

---

## 一、核心理念

### 1.1 角色分工（7 Agent 全激活）

| 角色 | 执行者 | 职责 | 激活阶段 |
|------|--------|------|----------|
| **产品指挥官（Simon）** | 用户 | 定义产品方向、审核关键决策、把控用户体验、客户视角验证 | 全程 |
| **Agent 调度中心** | SOLO 主对话 | 管理子 Agent 分工、信息流转、任务拆分、进度追踪、记忆管理 | 全程 |
| **① 产品经理 Agent** | Plan 子 Agent | 需求分析、技术方案设计、任务拆解、验收标准定义 | 全程 |
| **② 前端开发 Agent** | general_purpose_task 子 Agent | React 组件开发、UI 实现、状态管理、API 对接 | 开发期（主力） |
| **③ 后端开发 Agent** | general_purpose_task 子 Agent | Netlify Functions 开发、API 设计、数据存储、第三方集成 | 开发期（主力） |
| **④ 测试工程师 Agent** | general_purpose_task 子 Agent | 功能测试、Bug 修复、回归验证、性能检查 | 开发期（主力） |
| **⑤ 内容营销 Agent** | general_purpose_task 子 Agent | 产品文案、推广内容、SEO优化、使用教程、社交媒体内容 | 上线后（主力） |
| **⑥ 增长实验 Agent** | general_purpose_task 子 Agent | 用户行为分析、转化漏斗优化、推荐裂变机制、A/B测试 | 有用户后（主力） |
| **⑦ 日常运营 Agent** | general_purpose_task 子 Agent | 社区内容审核、用户反馈收集、数据周报、版本更新日志 | 社区期（主力） |

③ 后端开发 Agent ★★★   ③ 后端 ★★           ③ 后端 ★             ③ 后端 ★
④ 测试工程师   ★★★   ④ 测试 ★★           ④ 测试 ★★            ④ 测试 ★★
① 产品经理     ★★☆   ① 产品 ★★           ① 产品 ★★            ① 产品 ★★

⑥ 增长实验     ☆☆☆   ⑥ 增长 ★☆           ⑥ 增长 ★★★          ⑥ 增长 ★★

### 1.3 与 Reviewly 模式的异同

| 维度 | Reviewly（Simon） | GoNow（我们） |
|------|-------------------|---------------|
| 人类角色 | 产品+架构+客户 | 产品指挥官+最终用户 |
| Agent 框架 | OpenClaw | SOLO Task 子 Agent |
| Agent 数量 | 5 个（产品/开发/市场/增长/运营） | 7 个（产品/前端/后端/测试/营销/增长/运营） |
| 协作方式 | 异步任务队列 | 同步调度+并行执行 |
| 产出物 | SaaS 产品 | Web 应用 + PWA |
| 差异化 | 无独立测试Agent | 专职测试Agent保障质量 |

### 1.4 Vibe Coding 方法论融入

```
Vibe Coding 核心：快速迭代 → 即时反馈 → 持续优化

具体实践：
1. 每个功能开发完立即测试验证（不积累技术债）
2. 先跑通主流程，再打磨细节（先能用，再好用）
3. 遇到问题立即修复，不跳过（测试驱动）
4. 每次开发前后更新记忆文件（保持上下文连续性）
5. 小步提交，频繁验证（降低风险）
```

## 二、Agent 协作流程

### 2.1 单个功能开发流程

```
用户（Simon）提出需求
        │
        ▼
┌─────────────────────┐
│  Agent 调度中心       │
│  1. 读取记忆文件      │
│  2. 分析需求范围      │
│  3. 拆分开发任务      │
└─────────┬───────────┘
          │
    ┌─────┴─────┐
    ▼           ▼
┌────────┐ ┌────────┐
│产品Agent│ │前端Agent│  ← 可并行
│技术方案 │ │UI开发  │
└────┬───┘ └────┬───┘
     │          │
     ▼          ▼
┌────────┐ ┌────────┐
│后端Agent│ │测试Agent│
│API开发  │ │功能测试│
└────┬───┘ └────┬───┘
     │          │
     └────┬─────┘
          ▼
┌─────────────────────┐
│  Agent 调度中心       │
│  1. 汇总测试结果      │
│  2. 更新记忆文件      │
│  3. 向用户汇报进度    │
└─────────────────────┘
```

### 2.2 信息流转规则

```
记忆文件（GONOW_MEMORY.md）
    ├── 项目状态（当前阶段、已完成功能、待办事项）
    ├── 技术决策记录（为什么选X不选Y）
    ├── 已知问题列表（Bug + 临时解决方案）
    ├── 文件结构索引（关键文件路径和用途）
    └── API 接口记录（已实现的接口列表）
```

**读取时机**：每次开发任务开始前
**更新时机**：每次开发任务完成后

### 2.3 并行策略

- **可并行**：前端页面开发 ↔ 后端 API 开发（有明确接口约定时）
- **需串行**：后端 API 完成 → 前端对接 → 测试验证
- **独立模块**：不同功能模块可分配给不同子 Agent 并行开发

### 2.4 接口契约规范

前后端并行开发前，必须先定义接口契约，存放于 `/workspace/gonow/docs/api-contracts.md`：

```markdown
## POST /api/chat
**用途**：AI 对话代理
**请求体**：{ message: string, sessionId?: string, context?: object }
**响应体**：{ success: boolean, data: { reply: string, tripData?: Trip }, error?: string }
**错误码**：400 参数缺失 | 429 请求超限 | 500 AI 服务异常
**Mock 数据**：前端开发期间使用 mock 数据对接
```

### 2.5 LLM API 配置策略

| 配置项 | 开发期 | 生产期 |
|--------|--------|--------|
| LLM 提供商 | Claude API (Anthropic) | Claude API (Anthropic) |
| API Key 管理 | `.env` 本地文件（不提交 git） | Netlify 环境变量 |
| 调用方式 | 通过 Netlify Functions 代理 | 同左 |
| 降级方案 | 返回预设模板行程 + 错误提示 | 同左 |
| Token 限制 | 单次对话 4096 tokens | 同左 |

**环境变量清单**（需在 `.env` 中配置）：
```
VITE_AMAP_KEY=          # 高德地图 JS API Key（前端使用）
AMAP_WEB_KEY=           # 高德 Web Service API Key（后端使用）
ANTHROPIC_API_KEY=      # Claude API Key（后端使用）
WEATHER_API_KEY=        # 和风天气 API Key（后端使用）
```

## 三、Agent 详细职责

### 3.1 产品经理 Agent（Plan Agent）

**触发条件**：新功能开发前、技术选型时、架构调整时

**输入**：用户需求 + PRD 文档 + 记忆文件
**输出**：
- 技术方案文档（组件结构、数据流、API 设计）
- 任务拆解清单（每个子任务的目标、输入、输出、依赖）
- 验收标准（什么算"做完了"）

**关键原则**：
- 方案必须具体到文件级别（哪些文件要创建/修改）
- 必须考虑与已有代码的兼容性
- 必须定义清晰的验收标准

### 3.2 前端开发 Agent（Frontend Agent）

**触发条件**：技术方案确认后

**输入**：技术方案 + 组件设计 + API 接口定义
**输出**：
- React 组件代码（.tsx 文件）
- 样式代码（Tailwind CSS）
- 状态管理逻辑（React hooks / Zustand）
- 类型定义（TypeScript interfaces）

**关键原则**：
- 使用 shadcn/ui 组件库，不自造轮子
- 移动端优先，响应式设计
- 组件粒度适中，单一职责
- 必须包含 loading / error / empty 状态处理

### 3.3 后端开发 Agent（Backend Agent）

**触发条件**：技术方案确认后（可与前端并行）

**输入**：技术方案 + API 设计 + 数据模型
**输出**：
- Netlify Functions 代码（/netlify/functions/*.js）
- API 路由定义
- 数据存储逻辑（Netlify KV / localStorage）
- 第三方 API 集成（高德、天气、LLM）

**关键原则**：
- 所有 API Key 通过环境变量管理
- 统一错误处理格式
- API 响应格式一致：{ success, data, error }
- 必须处理边界情况（空数据、超时、格式异常）

### 3.4 测试工程师 Agent（Test Agent）

**触发条件**：代码开发完成后

**输入**：开发完成的代码 + 验收标准
**输出**：
- 测试报告（通过/失败/待修复）
- Bug 描述和复现步骤
- 修复建议

**关键原则**：
- 先验证主流程（Happy Path）
- 再验证边界情况
- 最后验证错误处理
- 发现问题 → 修复 → 重新测试 → 直到通过

### 3.5 内容营销 Agent（Marketing Agent）

**触发条件**：产品有可展示的 Demo 后即可启动，V1.0 上线后为主力

**输入**：产品功能说明 + PRD + 竞品分析 + 目标用户画像
**输出**：
- 产品介绍页文案（Landing Page）
- SEO 优化方案（meta 标签、structured data、sitemap）
- 社交媒体推广内容（小红书/公众号/知乎）
- 使用教程和帮助文档
- 版本更新公告（Release Notes）

**当前阶段任务（开发期轻量参与）**：
- 提前规划 Landing Page 文案结构
- 准备 SEO 基础配置（meta tags、Open Graph）
- 撰写产品介绍文案草稿

**关键原则**：
- 文案风格：简洁有力，突出核心卖点（红黑榜 + 故事化美食 + 景点多方案）
- 目标用户语言：说旅行者听得懂的话，不说技术术语
- 多平台适配：小红书（种草风）、公众号（深度文）、知乎（专业回答）

### 3.6 增长实验 Agent（Growth Agent）

**触发条件**：产品上线并有首批用户后激活

**输入**：用户行为数据 + 产品功能列表 + 竞品增长策略
**输出**：
- 用户转化漏斗分析报告
- A/B 测试方案（落地页 variants、注册流程优化）
- 裂变传播机制设计（分享得积分、邀请奖励）
- 留存策略（行程提醒推送、个性化推荐）
- 增长指标看板（DAU/MAU/留存率/分享率）

**当前阶段任务（开发期轻量参与）**：
- 设计分享裂变机制的技术方案（分享链接 → 注册 → 双方奖励）
- 规划用户行为埋点方案（哪些事件需要追踪）
- 设计增长指标体系

**关键原则**：
- 数据驱动，不靠直觉
- 每次只实验一个变量
- 小步快跑，快速验证
- 关注「啊哈时刻」：用户第一次感受到产品价值的瞬间

### 3.7 日常运营 Agent（Operations Agent）

**触发条件**：社区功能上线后为主力，开发期可轻量参与

**输入**：用户反馈 + 社区数据 + 产品使用数据
**输出**：
- 用户反馈周报（汇总 + 分类 + 优先级排序）
- 社区内容审核报告
- 数据周报（核心指标变化趋势）
- 版本更新日志
- FAQ 维护和更新

**当前阶段任务（开发期轻量参与）**：
- 建立 FAQ 文档框架（基于 PRD 中的常见问题）
- 设计用户反馈收集机制（应用内反馈入口）
- 规划数据埋点和分析方案

**关键原则**：
- 用户反馈 24 小时内响应
- 好的反馈和差的反馈同样重视
- 数据报告要有洞察，不只是数字堆砌
- 版本更新日志要让用户感受到产品在进步

### 5.3 更新规则
- **每次开发前**：读取记忆文件，了解当前状态
- **每次开发后**：更新记忆文件，记录进度和变更
- **遇到问题时**：记录到已知问题列表
- **做技术决策时**：记录决策和原因

## 六、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 子 Agent 上下文丢失 | 高 | 中 | 记忆文件 + 任务描述包含充分上下文 |
| 前后端接口不一致 | 中 | 高 | 先定义接口契约，再并行开发 |
| 生成代码质量不稳定 | 中 | 中 | 测试 Agent 兜底 + 人工审核关键模块 |
| 项目复杂度超预期 | 中 | 高 | 严格按 Phase 拆分，每个 Phase 独立可用 |
| 第三方 API 限制 | 低 | 中 | 提前了解免费额度，设计降级方案 |
